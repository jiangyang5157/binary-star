## CRITICAL Findings (Fix Before Any Live Trading)

### C4. Order Entry — Pivot Creates Naked Position Window
**File:** `src/agent/order_executor.py:140-176`  
**Severity:** CRITICAL — Financial loss risk  

The "Pivot-Preserve" case cancels ALL orders (including the existing position's OCO stop-loss protection), then places a new OCO. Between cancel and re-place, the position is **completely naked** — no stop loss. A violent market move in that sub-second window causes unlimited loss.

**Fix:** Place the new OCO FIRST, then cancel the old one. Or use OCO modification if the API supports it.

---

### C5. `MarginClient` — Wrong `sideEffectType` on SELL Orders
**File:** `src/infrastructure/binance/margin_client.py:195, 214`  
**Severity:** CRITICAL — Orders may be rejected  

```python
sideEffectType="MARGIN_BUY"  # Hardcoded regardless of side
```

For SELL orders (closing longs or opening shorts), `MARGIN_BUY` is semantically wrong. The Binance API expects `AUTO_REPAY` for sells (repaying borrowed assets). This could cause order rejections in production.

### H3. Volatility Normalization Uses Linear Instead of √Time Scaling
**File:** `src/analyzer/market_observer.py:441`  
**Severity:** HIGH — Chaos detection impaired  

```python
ratio = interval_seconds(macro) / interval_seconds(micro)  # e.g., 3600/900 = 4
volatility_expansion_index = atr_n / (atr_m / ratio)  # Divides by 4
```

Volatility scales with **√time**, not linearly. With 1h vs 15m, ATR should be normalized by √4 = 2, not 4. The code **understates** volatility expansion by ~2× in this case, meaning the system fails to detect chaos regimes when it should.

---

### H4. `project_holding_time` — Division-by-Zero Risk
**File:** `src/utils/math_utils.py:246-252`  
**Severity:** HIGH — Crash in edge case  

`effective_velocity = effective_velocity_per_atr * atr`. If both `min_velocity_floor=0` and `normalized_velocity=0`, `effective_velocity_per_atr = max(0, 0, 1e-9) = 1e-9`. With tiny ATR (stablecoin), `effective_velocity` could be 0 in float arithmetic. Add explicit guard.

## Prompt & Strategy Logic Review

### Prompt Analysis — `binary_star.md` (Shared System Preamble)

**Findings:**
- **P1.** `IS_SQUEEZING` uses `squeeze_threshold` (1.0 from config) but `IS_CHAOS` uses `volatility_extreme_ratio` (2.2). These are correctly templated — no bug here, but the thresholds should be reviewed for consistency (squeeze < 1.0 triggers more easily than chaos > 2.2).
- **P2.** The LOGIC_MACROS section defines `HAS_ABSORPTION_RISK` without templating variables — it's hardcoded as `oi_delta_micro < 0 AND abs(cvd_intensity_ratio) > {cvd_intensity_extreme}`. The `cvd_intensity_extreme` variable IS templated correctly.
- **P3.** The visual chart description (section 5) is extremely detailed. With ~500 words describing chart layout, this consumes significant context tokens. Consider moving to a concise reference.

### Prompt Analysis — `session.md` (Session Analyst)

**Findings:**
- **P4.** **CRITICAL LOGIC DIVERGENCE:** Line 55 instructs the agent to manually calculate `projected_holding_hours = abs(take_profit - entry) / atr_macro * unit_atr_holding_hours`. But `unit_atr_holding_hours` was injected by `_inject_regime_benchmarks()` with dilation already applied. Then `MathFactChecker.verify()` ALSO calculates `project_holding_time()` using the Python `project_holding_time()` function. In `_finalize_and_sanitize()`, the Python-verified value OVERWRITES the agent's manual calculation. So the agent's calculation is **always discarded** — but the agent spends reasoning tokens on it. The prompt should explicitly say "calculate a rough estimate; the system will replace it with the Python-verified value."
- **P5.** The CONFIDENCE CALCULUS (lines 100-112) is a 4-dimension penalty system with max 30+30+20+20=100. The Critic's Crucible dimension awards 0 points during planning or if any CONSTRUCTIVE/TERMINAL challenge remains. This means **every first-round plan has max possible score of 80** (since Critic's Crucible = 0). The prompt doesn't explicitly state this, which may confuse the LLM.
- **P6.** "TOOL ERROR FALLBACK: If MathTools returns an error, immediately abort to NEUTRAL" (line 25) — this is aggressive. A transient API error in the math tool causes trade abortion. Should allow 1 retry.
- **P7.** The SHIELD LAW (line 48) says "anchor MUST sit strictly BETWEEN entry and stop_loss." But the MOMENTUM EXEMPTION (line 50) relaxes this when `IS_TREND_STRONG` and POC > `poc_gravity_atr_distance` (3.5 ATR). This exemption is reasonable but creates an asymmetry: in strong trends with distant POC, the anchor requirement is waived, potentially allowing stops in liquidity vacuums.

### Prompt Analysis — `critic.md` (Critic Agent)

**Findings:**
- **P8.** The CRITIC_CODES table has an **implicit priority inversion**: `[FLOW_VIOLATION]` (fighting CVD flow direction) is CONSTRUCTIVE, but `[RETAIL_LONG_SQUEEZE]` / `[RETAIL_SHORT_SQUEEZE]` (fighting retail positioning) is TERMINAL. This means the system treats "retail is crowded" as a harder veto than "CVD flow is opposite." This is a deliberate design choice (retail positioning is a contrarian indicator) but the asymmetry should be explicitly documented.
- **P9.** `IS_OVEREXTENDING` definition (line 38) correctly uses: `NOT (IS_TREND_STRONG AND HAS_FLOW_DOMINANCE)` as the exemption. This allows aggressive trend-following entries far from POC when both trend AND flow confirm.
- **P10.** `IS_HOLDING_TOO_LONG` (line 39) uses `max_holding_hours * temporal_weight_factor` from the math verification. But `temporal_weight_factor` comes from the regime scalars which are computed from the **current** regime. If the regime changes mid-trade, the threshold changes but the elapsed time doesn't reset. This is actually reasonable behavior.
- **P11.** THE NEUTRALITY PARADOX protocol is well-designed — it prevents the Critic from endlessly challenging NEUTRAL decisions when structural reasons justify inaction.

### Prompt Analysis — `evolver.md` (Meta-Evolver)

**Findings:**
- **P12.** `IS_OVERFIT_RISK`: "Historical fix would invalidate > 5% of Pristine success records." The LLM has **no way to actually validate this** — it can't run backtests. This is a heuristic guideline that the LLM must approximate from qualitative analysis of the audit reports. The Evolver may overfit or underfit based on LLM hallucination of the "5% threshold."
- **P13.** The `SEMANTIC_REFINEMENT` anchor rule (line 102) requires "character-for-character, byte-perfect copy" of prompt text. If the prompts have been modified since the audit was generated (e.g., by a previous evolution run), the EVOLVER's anchor won't match and the patch will silently fail with "NO MATCH."
- **P14.** The `DE_SENSITIZATION` deadlock-breaking action is triggered when `max_rounds` was exceeded. Current `max_rounds=2`, which means 2 rounds with TERMINAL/CONSTRUCTIVE veto = deadlock. With only 2 rounds configured, deadlocks reach cold synthesis quickly. The evolvers' DE_SENSITIZATION path is unlikely to ever trigger.

### Prompt→Code Integration Issues

- **P15.** Session agent is told to use `unit_atr_holding_hours` for time projection (session.md line 55), which is injected by `_inject_regime_benchmarks()`. But this function only runs ONCE at the start of `execute_flow()`. If the debate goes multiple rounds and the market regime changes between rounds, the injected `temporal_physics` in the observation is stale. The agent would use stale velocity estimates.
- **P16.** The critic prompt's `HAS_ANCHOR_VIOLATION` check references `near liquidation_cluster AND (stop_loss >= cluster OR stop_loss <= cluster)` but the observation JSON's `liquidation_clusters` structure is always empty (Binance API has no public liquidation data endpoint). This entire check path is a no-op.

## Configuration & YAML Review

### `strategy_config.yaml`

- **CFG1.** `volume_profile_value_area_width: 0.86` — 86% Value Area is unusually wide (standard is 68-70%). This captures more volume but makes VAH/VAL boundaries less significant as support/resistance.
- **CFG2.** `squeeze_threshold: 1.0` — BB_width / KC_width < 1.0 means Bollinger Bands are inside Keltner Channels. This is the standard definition. The `squeeze_audit_threshold: 0.85` is a separate, tighter threshold used only by the Critic's fast-pass check.
- **CFG3.** `min_rr_ranging: 1` and `min_rr_trending: 1.2` — Ranging allows 1:1 RR (50% win rate = breakeven). Trending requires 1.2:1. With `chaos_rr_discount: 0.35`, chaos minimum becomes 0.65 for ranging and 0.78 for trending — appropriately aggressive for survival mode.
- **CFG4.** `max_holding_hours: 72.0` with `temporal_weight_climax: 0.3` → effective max hold in climax = 21.6 hours. With `time_stop_multiplier: 1.5` → time-stop at 32.4 hours. Reasonable.
- **CFG5.** `wick_skew_fallback: 0.5` — When a candle has zero range (doji), the fallback assumes neutral wick skew. A doji at a boundary is actually a significant signal — a more informative fallback would look at the prior candle or use `None`.

### `global_config.yaml`

- **CFG6.** `active_provider: "deepseek"` — DeepSeek is used as the primary LLM. DeepSeek does NOT support vision (confirmed by `supports_vision: false` in the config). This means chart images (VisualPart) are **never sent to the LLM**. The prompt instructions reference `[VISUAL_CONTEXT]` extensively, but the agent never actually sees the charts. **This is a significant strategy gap** — the Session and Critic agents are making decisions without the visual ground truth they're instructed to use.
- **CFG7.** `max_rounds: 2` — Only 2 debate rounds. This means: Round 1 (planning + audit), Round 2 (revision + audit). If Round 2 fails, cold synthesis produces the final decision. This is tight but cost-efficient. The risk is that complex market setups need 3+ rounds to converge.
- **CFG8.** `risk_per_trade: 0.004` (0.4% of equity). Conservative. With 0.4% risk per trade and a 1.2 RR minimum, the breakeven win rate is ~45%.
- **CFG9.** `net_qty_tolerance: 0.00001` — This is the minimum BTC quantity to consider a position "FLAT." At $100k/BTC, this is $1. Reasonable, but if BTC price falls to $10k, this becomes $0.10 which might be too tight (rounding errors could show phantom positions).
- **CFG10.** Sniper `pulse_interval_minutes: 2.0` with `pulse_cooldown_multiplier: 2.5` → cooldown = 5 minutes. With `chaos_cooldown_multiplier: 2.0` → chaos cooldown = 10 minutes. The `state_lockout_hours: 8.0` blocks retriggers for a full trading session.
- **CFG11.** Guardian trailing levels: Level 1 at 1.5 ATR profit (SL→breakeven), Level 2 at 2.5 ATR (SL→entry+0.5 ATR), Level 3 at 4.0 ATR (SL→entry+1.5 ATR). These are aggressive — a position needs to move 1.5× ATR before even reaching breakeven. Many trades will be stopped out before trailing activates.

### `symbol_config.yaml`

- **CFG12.** BTCUSDT `sl_slippage_buffer: 10.0` — A $10 buffer on BTC stop-loss. At $100k BTC, this is 0.01% — very tight. During volatility, this could cause the stop limit to not fill.
- **CFG13.** XAUTUSDT `sl_slippage_buffer: 0.5` — $0.50 buffer on gold. At ~$3000 XAUT, this is 0.017% — similarly tight.
- **CFG14.** XAUTUSDT overrides: `cvd_divergence_tick_delta: 0.18` and `cvd_impulse_tick_delta: 0.25` (lower than BTC defaults of 0.22/0.35). This makes XAUT more sensitive to flow signals — appropriate for a thinner market.

## Architecture & Strategy Observations

### Strengths

1. **Dependency Injection Everywhere** — `AbstractAIClient`, `AbstractExchangeClient`, config dataclasses all injected. Clean testing seams.
2. **Frozen Dataclasses for Config** — Immutable configuration prevents accidental mutation.
3. **Domain Exception Hierarchy** — `SingularityError → AgentInferenceError → ...` enables precise error handling.
4. **Adversarial Debate Architecture** — Session→Critic with MathFactChecker verification. Genuinely innovative.
5. **Guardian System** — Progressive trailing stop + time-based stop + emergency fallback. Professionally designed.
6. **Provider-Agnostic AI Layer** — `VisualPart`, `AIResponse`, `ToolCall` abstractions decouple agents from SDKs.
7. **Config Resolution Stack** — `base + symbol.overrides → resolved` is clean and testable.

### Architecture Risks

1. **Visual Context Disabled:** DeepSeek (`supports_vision: false`) means chart images are NEVER sent to the LLM. The Session and Critic prompts extensively reference `[VISUAL_CONTEXT]` but the agents operate text-only. The system is effectively running blind to the charts it generates.
2. **Single Point of Failure: AI Inference.** No fallback to deterministic strategy on provider outage.
3. **No Paper Trading / Fast Backtest Mode.** All backtesting goes through the full LLM pipeline, incurring API costs.
4. **Configuration Complexity:** ~150+ parameters across 4 YAML files. A single misconfigured value could cause financial loss.
5. **Binary Star Protocol Assumes Rational Debate.** If both agents hallucinate in the same direction, the adversarial safeguard fails. MathFactChecker only verifies arithmetic, not directional correctness.
6. **No Account-Level Drawdown Circuit Breaker.** The strategy intent says "LARGE DRAWDOWNS ARE ABSOLUTELY PROHIBITED" but the system has no mechanism to detect or enforce this.
